### Tier 2: Design choice — consolidate permissive policies for scale

`multiple_permissive_policies` × 14 across `audit_log`, `callbacks`,
`lead_events`, `leads`, `user_roles`. Today each role has its own
permissive policy per `(table, action)`; supabase-js evaluates every
permissive policy for every row. Consolidating into a single policy per
(table, action) with `OR`'d `USING` clauses cuts per-row eval cost by
the role count without changing semantics.

This is **risky to ship in a hurry** — a wrong consolidation = a real
RLS leak. Defer to a future session with the cross-tenant integration
test suite re-run before and after. Migration drafts only the Tier 1
fixes; Tier 2 needs its own plan + tests.

### Tier 4: By design — accept the WARN

`authenticated_security_definer_function_executable` × 11 functions
(`agent_performance_in_range`, `agent_stats_in_range`, `complete_call`,
`country_stats_in_range`, `group_speed_to_lead_series`,
`mark_lead_contacted`, `record_audit`, `record_no_answer`,
`reassign_lead`, `schedule_callback`, `speed_to_lead_series`).

These are the Phase 2-5 architectural pattern: SECURITY DEFINER RPCs
called from authenticated cookie sessions, with internal role/country
guards. The lint cannot verify the internal guards. Switching to
SECURITY INVOKER would require collapsing every RPC's internal logic
into RLS-friendly SQL — a much larger refactor for no real security
gain (the guards are tested in vitest cross-tenant suites).

Accepted; documented inline in the migration comment so the next
auditor sees the rationale.
