  <name>Task 1: Migration 00011 — 4 country-admin views (security_invoker)</name>
  <files>packages/supabase/migrations/00011_country_admin.sql</files>
  <action>
Create `packages/supabase/migrations/00011_country_admin.sql`. Define four views, each ending with `WITH (security_invoker = true)` so RLS on `leads` / `lead_events` does the country lock. Grant `SELECT` on each view to `authenticated`.

1. **`country_today_stats`** — one row per active country with the five KPI tile counts AND the "vs yesterday" deltas. Columns:
   - `country_code text` (FK to `countries.code`)
   - `total_leads bigint` (all-time count where `country_code` matches)
   - `new_today bigint` (`status='new' AND created_at >= start_of_today_in_country_tz`)
   - `contacted_today bigint` (`status='contacted' AND first_contacted_at >= start_of_today_in_country_tz`)
   - `converted_today bigint` (`status='converted' AND updated_at >= start_of_today_in_country_tz`)
   - `lost_today bigint` (`status='lost' AND updated_at >= start_of_today_in_country_tz`)
   - `new_yesterday bigint`, `contacted_yesterday bigint`, `converted_yesterday bigint`, `lost_yesterday bigint` (same predicates against the yesterday calendar day in country tz)
   - **Time zone:** "today" = `(now() AT TIME ZONE c.timezone)::date` per `countries.timezone` from migration 00004 reference data. Use `date_trunc('day', now() AT TIME ZONE c.timezone) AT TIME ZONE c.timezone` to get a UTC timestamptz boundary. LEFT JOIN from `countries` so every active country gets a row even with zero leads (UI doesn't have to handle missing rows — same shape as `agent_today_stats` from migration 00009).

2. **`leads_by_service_today`** — counts per `form_slug` per `country_code` for today (calendar day in country tz). Columns: `country_code text`, `form_slug text`, `leads_count bigint`. One row per (country, service). UI sorts and renders the top N.

3. **`status_pipeline_today`** — funnel counts per `country_code` per status, today. Columns: `country_code text`, `status lead_status`, `count bigint`. Status values = the full `lead_status` enum (`new`, `contacted`, `qualified`, `converted`, `lost`). Yes, `qualified` is included even though Phase 3 plan 03-04 rejected it from `complete_call` — the enum value is preserved for analytics back-compat (per STATE.md). UI renders the funnel as five segments.

4. **`country_speed_to_lead_today`** — gauge data per country for today. Columns:
   - `country_code text`
   - `total_contacted bigint` (today's leads with `first_contacted_at IS NOT NULL`)
   - `on_target_count bigint` (today's leads where `first_contacted_at - created_at <= interval '5 minutes'`)
   - `on_target_pct numeric(5,2)` (`on_target_count * 100.0 / NULLIF(total_contacted, 0)`)
   - `avg_response_seconds numeric(10,2)` (avg over `EXTRACT(EPOCH FROM first_contacted_at - created_at)` where `first_contacted_at IS NOT NULL`)

   Document the NULL policy in a SQL comment: speed-to-lead aggregations operate over leads with `first_contacted_at IS NOT NULL` to avoid the "artificially fast" pitfall (Phase 4 RESEARCH.md pitfall 3).

**Reuse vs. new:** Migration 00006 already ships `speed_to_lead_daily` (per-country per-day P50/P95). Phase 4 creates `country_speed_to_lead_today` as a today-only single-row-per-country variant for the gauge tile — different shape, both kept. Note this in the SQL comment.

**RLS:** Underlying `leads` / `lead_events` tables already enforce `country_code = (auth.jwt() ->> 'country_code')` for `country_admin`, allow-all for `hq_admin`. Views don't need their own policies — `security_invoker = true` is sufficient.

Add a header comment block at the top of the file naming the migration: `-- 00011_country_admin: views + RPCs for the Country Admin Dashboard (Phase 4)`.

**Apply via MCP:** Use `mcp__supabase-paratusgroup__apply_migration` with `name: "country_admin"` and the file's body. Verify via `mcp__supabase-paratusgroup__list_migrations`.

<task type="auto">
  <name>Task 2: Migration 00011 — 4 country-admin RPCs (SECURITY DEFINER + JWT guards)</name>
  <files>packages/supabase/migrations/00011_country_admin.sql</files>
  <action>
Append four RPCs to `00011_country_admin.sql`. Mirror the shape of `mark_lead_contacted` (migration 00009) — `SECURITY DEFINER`, `SET search_path = public, pg_temp`, JWT country + role guards inside the function, `REVOKE ALL ... FROM public, anon`, `GRANT EXECUTE ... TO authenticated`.

1. **`country_stats_in_range(p_country text, p_from timestamptz, p_to timestamptz) RETURNS TABLE (converted_count bigint, lost_count bigint, contacted_count bigint, new_count bigint)`**
   - JWT guard: `country_admin` must have `auth.jwt() ->> 'country_code' = p_country`; `hq_admin` allowed any country; reject otherwise with `'forbidden_country'` (`ERRCODE = '42501'`).
   - Returns the four range-aware status counts for the date range. Counted by `status` and `updated_at` in `[p_from, p_to)` for `converted` / `lost`; by `created_at` in range for `new`; by `first_contacted_at` in range for `contacted`.
   - Same shape pattern as `agent_stats_in_range` from migration 00010 (single-row return).

2. **`agent_performance_in_range(p_country text, p_from timestamptz, p_to timestamptz) RETURNS TABLE (agent_id uuid, full_name text, leads_assigned bigint, leads_contacted bigint, leads_converted bigint, leads_lost bigint, avg_response_seconds numeric(10,2))`**
   - Same JWT guard as above.
   - One row per agent in the country. LEFT JOIN from `user_roles` filtered to `role='sales_rep'` and `country_code=p_country` — so inactive/empty agents still show up (same precedent as `agent_today_stats`).
   - `avg_response_seconds`: `AVG(EXTRACT(EPOCH FROM first_contacted_at - created_at)) FILTER (WHERE first_contacted_at IS NOT NULL AND created_at >= p_from AND created_at < p_to)`.
   - Driver of the Sales Rep Performance table.

3. **`speed_to_lead_series(p_country text, p_from timestamptz, p_to timestamptz) RETURNS TABLE (day date, median_seconds numeric(10,2), p75_seconds numeric(10,2))`**
   - Same JWT guard.
   - One row per `created_at::date` (in country tz) inside `[p_from, p_to)`. Median and P75 of `EXTRACT(EPOCH FROM first_contacted_at - created_at)` where `first_contacted_at IS NOT NULL`.
   - Use `percentile_cont(0.5) WITHIN GROUP (ORDER BY ...)` and `percentile_cont(0.75) WITHIN GROUP (ORDER BY ...)`.
   - **Decision (per RESEARCH.md open question 1):** sparkline uses `median_seconds` (resists outliers); the KPI tile labelled "Avg Response Time" still reads from `country_speed_to_lead_today.avg_response_seconds`. The asymmetry is intentional — document with a SQL comment on the function.

4. **`reassign_lead(p_lead_id uuid, p_to_agent_id uuid) RETURNS void`**
   - JWT role guard: `auth.jwt() ->> 'role' IN ('country_admin','hq_admin')`; reject otherwise with `'forbidden_role'`.
   - Lookup `v_lead_country := (SELECT country_code FROM leads WHERE id = p_lead_id)`. NULL → raise `'not_found'` (`ERRCODE = 'P0002'`).
   - Lookup `v_target_country := (SELECT country_code FROM user_roles WHERE user_id = p_to_agent_id)`. NULL → raise `'not_found'`.
   - Country-scope guard: if role is `country_admin` and `auth.jwt() ->> 'country_code' <> v_lead_country` → raise `'forbidden_country'`.
   - **Cross-country guard (defence-in-depth):** if `v_target_country <> v_lead_country` → raise `'cross_country_assignment'` (`ERRCODE = '42501'`). This stops a country admin from re-assigning to a different country's agent even though the JWT-country guard would have already blocked the parent reassignment intent.
   - Mutation: `UPDATE leads SET assigned_to = p_to_agent_id, updated_at = now() WHERE id = p_lead_id`. Insert audit row: `INSERT INTO lead_events (lead_id, actor_id, type, payload, country_code) VALUES (p_lead_id, (SELECT auth.uid()), 'reassigned', jsonb_build_object('to_agent_id', p_to_agent_id), v_lead_country)`. The `country_code` column on `lead_events` is denormalised from `leads` per Phase 2 decision; the BEFORE INSERT trigger from migration 00005 backstops it but we set explicitly here for clarity.

For each function:
```
REVOKE ALL ON FUNCTION public.<name>(<args>) FROM public, anon;
GRANT EXECUTE ON FUNCTION public.<name>(<args>) TO authenticated;
```

Apply via `mcp__supabase-paratusgroup__apply_migration`.
