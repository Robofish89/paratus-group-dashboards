    1. **Migration `packages/supabase/migrations/00018_leads_cursor_index.sql`** — apply via `mcp__supabase-paratusgroup__apply_migration`:
       ```sql
       -- Composite index matching the keyset pagination sort key (created_at DESC, id DESC).
       -- Phase 4 used offset pagination; Phase 6 migrates to cursor for asymptotic correctness.
       -- The DESC clauses match the query's ORDER BY exactly so Postgres can skip-scan.
       CREATE INDEX leads_created_at_id_desc_idx ON leads (created_at DESC, id DESC);
       ```
       Migration header explains: this is a v2 index that *augments* (does not replace) any single-column index on `created_at`; Phase 4's offset queries continue to work; Phase 6 cursor queries take advantage of the composite for the tuple-compare lookup.

    2. **Cursor helper at `apps/web/app/(country-admin)/[country]/_lib/leads-cursor.ts`**:
       - Type `LeadCursor = { created_at: string; id: string }` (ISO timestamp + UUID).
       - `encode(cursor: LeadCursor): string` — `Buffer.from(JSON.stringify(cursor)).toString('base64url')`.
       - `decode(s: string | undefined): LeadCursor | null` — base64url decode + JSON.parse + zod validate (return null on any error rather than throwing — invalid cursor = "go to first page").
       - Use `base64url` encoding (URL-safe, no `=` padding) so cursors survive copy-paste in URLs without escaping.

    3. **Page at `apps/web/app/(country-admin)/[country]/leads/page.tsx`** — modify the existing offset query to keyset:
       ```ts
       const cursor = decode(searchParams.cursor);
       let q = supabase
         .from('leads')
         .select('*, assigned_user:user_roles!leads_assigned_to_fkey(user_id, full_name)', { count: 'exact' })
         .eq('country_code', countryCode)
         .order('created_at', { ascending: false })
         .order('id',         { ascending: false })
         .limit(PAGE_SIZE + 1);  // fetch one extra to detect hasMore
       if (cursor) {
         // Tuple compare: (created_at, id) < (cursor.created_at, cursor.id)
         q = q.or(
           `created_at.lt.${cursor.created_at},and(created_at.eq.${cursor.created_at},id.lt.${cursor.id})`
         );
       }
       const { data: rows, count } = await q;
       const hasMore = rows.length > PAGE_SIZE;
       const visible = hasMore ? rows.slice(0, PAGE_SIZE) : rows;
       const nextCursor = hasMore ? encode({ created_at: visible.at(-1)!.created_at, id: visible.at(-1)!.id }) : null;
       ```
       — Replace the prev/next pagination component to use `?cursor=...` instead of `?page=N`. Keep the existing search/filter UI untouched (filters apply BEFORE the cursor — order is sort-then-paginate-by-cursor).
       — `count: 'exact'` is still useful for the total-count display; if it tanks performance at scale it can drop to `'estimated'` later (track in 06-05 RUNBOOK).
       — Sanitise the existing `?q=` filter the same way Phase 4 plan 04-03 already did (strip `,()` characters before `.or()`).
       — Update the page-state URL contract: `?cursor=<base64>&q=<text>&filter=<status>` instead of `?page=N`.

    AVOID: keeping the offset path as a fallback (dead code paths rot — cut over fully). AVOID encoding non-deterministic data in the cursor (no timestamps from `now()`, no random ids). AVOID URL-encoding the base64 manually — `base64url` already produces URL-safe output.
