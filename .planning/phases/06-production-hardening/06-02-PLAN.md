<task type="auto">
  <name>Task 2: Rate limiting + security headers</name>
  <files>packages/supabase/src/lib/rate-limit.ts, apps/web/middleware.ts, apps/web/next.config.ts, apps/web/app/api/leads/ingest/route.ts</files>
  <action>
    1. `packages/supabase/src/lib/rate-limit.ts` — thin wrapper around `@upstash/ratelimit` + `@upstash/redis`. Export named limiters:
       - `authLimiter` — 5 req / 60s per IP
       - `ingestLimiter` — 60 req / 60s per shared-secret hash (so legit Paratus traffic isn't IP-throttled, but a leaked secret is bounded)
       - Each returns `{ success, limit, remaining, reset }`
       - `import 'server-only'`
       - Fail-open in dev (`if (!process.env.UPSTASH_REDIS_REST_URL) return { success: true, ... }`) so local development without Upstash still works; fail-closed in prod (`throw` if env missing in `NODE_ENV === 'production'`).
    2. `apps/web/middleware.ts` — apply `authLimiter` to `/login`, `/auth/callback`, `/auth/reset` and any `POST` to `/api/auth/*`. On rate-limit, return 429 with `Retry-After` header set to `Math.ceil((reset - Date.now())/1000)`.
    3. `apps/web/app/api/leads/ingest/route.ts` — apply `ingestLimiter` keyed on `sha256(LEAD_INGEST_SHARED_SECRET)` BEFORE the secret check (so rate-limit headers are stable even on 401s). Return 429 on breach.
    4. `apps/web/next.config.ts` — add `headers()` exporting:
       - `Content-Security-Policy`: `default-src 'self'; script-src 'self' 'unsafe-inline' va.vercel-scripts.com; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; connect-src 'self' *.supabase.co wss://*.supabase.co https://api.resend.com; frame-ancestors 'none'` (tighten over time; document in SECURITY_CHECKLIST)
       - `Strict-Transport-Security`: `max-age=63072000; includeSubDomains; preload`
       - `X-Frame-Options`: `DENY`
       - `X-Content-Type-Options`: `nosniff`
       - `Referrer-Policy`: `strict-origin-when-cross-origin`
       - `Permissions-Policy`: `camera=(), microphone=(), geolocation=()`
    5. Coordinates with Plan 01: Plan 01 added `LEAD_INGEST_SHARED_SECRET` validation; Plan 02 layers rate-limit BEFORE the secret check. No file conflict — Plan 02 edits a different region of the same handler. Run sequentially within Wave 1 for the ingest file (lock via depends-on at execute-phase orchestrator level if needed; in practice, the merge order here is: Plan 01 lands first within this wave, then Plan 02 layers on top).

    Note on file conflict: `apps/web/app/api/leads/ingest/route.ts` is touched by both Plan 01 and Plan 02. The orchestrator must run Plan 01 first inside Wave 1 (fastest path: run them serially, accept the parallelism loss on this file). Marking Plan 02's touch as additive: Plan 02 only prepends a `await ingestLimiter.limit(...)` block; no logic Plan 01 wrote is rewritten.

    AVOID: in-memory rate-limit (resets per Vercel cold start = useless). MUST use Upstash.
  </action>
