<objective>
Ship the Lead List surface and the two write APIs that power it: `/api/country-admin/reassign` (calls `reassign_lead` RPC) and `/api/country-admin/export-leads` (CSV stream via `Papa.unparse`, cookie-authed so RLS country-locks the rows). Build the filterable, paginated lead list page at `/[country]/leads` with a reassign dialog driven by `getCountryAgents`.

Purpose: Closes the country-admin write surface — admins can re-route a stuck lead to another agent in their country, and pull a CSV when stakeholders want offline analysis. Reuses the SECURITY DEFINER + JWT-guard pattern locked in Phase 3 and the papaparse precedent locked in Phase 2 plan 02-05.
Output: An NA country admin can navigate Sidebar → Leads, filter by status/service/date/search, click "Reassign" on any row, pick a target agent, save, and see the row update. They can click "Export CSV" and get a downloaded file scoped to their country.
</objective>

**1. `POST /api/country-admin/reassign`** (`runtime = "nodejs"`):
- Parse `{ lead_id, to_agent_id }` with the `ReassignLeadInput` Zod schema from `@repo/supabase/schemas`. Invalid → `400 invalid_payload`.
- Call `reassignLead(...)` from `@repo/supabase/dal/country` (cookie-authed; RLS in force; the SECURITY DEFINER RPC does the role + country + cross-country guards).
- Map errors:
  - `ForbiddenError` → `403 forbidden`
  - `NotFoundError` → `404 not_found`
  - any other → `500 internal_error` (log via `console.error` with structured payload)
- Success → `204 No Content`.
- Mirror the shape of `apps/web/app/api/queue/complete/route.ts` exactly — same error mapping, same Zod parse pattern, same nodejs runtime.

**Auth note:** middleware already enforces cookie session for `/api/country-admin/*` (the public-paths block in `apps/web/middleware.ts` only opens `/api/leads/*` for HMAC). The route handler additionally relies on `reassign_lead`'s internal JWT role guard for defence-in-depth.

**2. `GET /api/country-admin/export-leads?from=&to=&status=&service=&q=`** (`runtime = "nodejs"`):
- **Use `createClient` from `@repo/supabase/server` — NOT `createAdminClient`** (RESEARCH.md pitfall 6). RLS is the country lock; a country admin physically cannot select rows from other countries.
- Read query params. Optional: `from` and `to` (ISO datetime), `status` (`'new' | 'contacted' | 'qualified' | 'converted' | 'lost'`), `service` (form_slug), `q` (full-text — applied to `full_name ILIKE %q% OR email ILIKE %q% OR phone_e164 ILIKE %q%`).
- Build the query against `leads`:
  - `.select("id, full_name, email, phone_e164, status, form_slug, assigned_to, country_code, created_at, first_contacted_at, lost_reason")`
  - `.order("created_at", { ascending: false })`
  - `.gte("created_at", from)` if `from`
  - `.lt("created_at", to)` if `to`
  - `.eq("status", status)` if `status`
  - `.eq("form_slug", service)` if `service`
  - `.or(...)` for the `q` filter if `q`
- Run query. On error → `500 {error.message}`.
- `const csv = Papa.unparse(data ?? [])` (papaparse already a dep from Phase 2; import as `import Papa from "papaparse"`).
- Build filename: `leads-${country_from_jwt}-${from ?? 'all'}-to-${to ?? 'now'}.csv`. Country read from the cookie session (`(await supabase.auth.getUser()).data.user?.app_metadata?.country_code`) — for HQ admin this is omitted and the filename is `leads-all-...`.
- Return:
```
return new Response(csv, {
  headers: {
    "Content-Type": "text/csv; charset=utf-8",
    "Content-Disposition": `attachment; filename="${filename}"`,
  },
});
```

**Cap the row count:** add `.limit(50000)` to prevent runaway exports. If exceeded, the country admin gets the first 50k rows with a `X-Truncated: true` response header — log a warning. (Phase 4 acceptance: Paratus's largest active country has ~5k leads to date; 50k headroom is 10× buffer.)

**Don't filter by country_code in the handler** — RLS does it. Do NOT `.eq("country_code", ...)` here; that would silently no-op for HQ admins (they need to see all countries) and is dead code for country admins (RLS already blocks others).
  </action>

- Read `?status=&service=&from=&to=&q=&page=` from `searchParams`. Default `page=1`, `pageSize=50` (mockup's pagination pattern).
- Fetch in parallel:
  - Filtered leads via Supabase query mirroring the export query in task 1 — same filters but with `.range((page-1)*50, page*50-1)` and `count: 'exact'`. Use cookie-authed `createClient`.
  - `getCountryAgents(country)` — drives the reassign dropdown.
- Compose `<DashboardLayout>` → page heading "Leads" + Export-CSV link button (`<a href="/api/country-admin/export-leads?...">`, anchor not button so the browser handles the file download natively) → `<LeadList>` (props: rows, total, page, pageSize, agents).

**Pagination decision:** offset-based for v1 (RESEARCH.md pitfall 8 + open question 3). Document the cursor-migration TODO in a Phase-6 carry-over comment in the SUMMARY (do NOT add a TODO in code per the Boil-the-Ocean standard — code is correct as-is for v1's traffic).

**Realtime decision:** **do NOT subscribe to broadcasts on the lead list** (RESEARCH.md pitfall 8). Pagination + realtime insertion creates index-shift bugs. The dashboard tiles still pop via `useCountryBroadcast` — agent reassignment is rare enough that the admin can refresh.

**2. `_components/lead-list.tsx`** ("use client" — for the filter form + reassign-dialog open state):
- Filter row above the table: status select (`<Select>`), service select, date-range picker (reuses `DateRangePicker` from `(sales-rep)/_components/date-range-picker.tsx` if it's exported there; if not, extract it to `apps/web/app/_components/date-range-picker.tsx` first since both surfaces use it now). Search input. Each filter pushes to URL via `router.push` with `useSearchParams`.
- Table columns: Name (with phone subline), Service, Status (`<StatusBadge>`), Assigned To (agent name or "Unassigned"), Created, Actions.
- Actions: a single ⋮ menu with "Reassign…" (opens dialog).
- Pagination: prev/next + page count, links update `?page=`.

**3. `_components/reassign-dialog.tsx`** ("use client") — wraps `<Dialog>` from `@repo/ui`:
- Props: `lead: { id, full_name, current_assignee }`, `agents: AgentOption[]`, `onClose`, `onReassigned`.
- UI: "Reassign {lead.full_name}" + agent select + Cancel + Save buttons.
- On save: `fetch("/api/country-admin/reassign", { method: "POST", body: JSON.stringify({ lead_id, to_agent_id }) })`.
- On `204`: call `onReassigned`, then `router.refresh()` to re-fetch the list (server-authoritative).
- On `403`: show inline error "You don't have permission to reassign this lead."
- On `404`: show "Lead no longer exists" (race: someone else might have deleted/moved).
- On `500` or network error: show "Couldn't reassign. Try again." Surface the error via `<Sonner>` toast (already wired in the layout).

**Cross-country reassignment is enforced at the RPC layer** — the agents dropdown is already filtered to the lead's country (it's `getCountryAgents(country)` from the page), and the RPC's cross-country guard backstops it. No client-side cross-country check.
