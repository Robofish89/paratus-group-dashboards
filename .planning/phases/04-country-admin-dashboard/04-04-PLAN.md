  <action>
Mirror `apps/web/e2e/sales-rep-golden-path.spec.ts` exactly for setup, env, teardown. Use the `/api/e2e-login` bridge (gated by `E2E_AUTH_ENABLED`, absent in production per STATE.md). Sign in as the NA country admin; seed two NA leads via the service-role client in `beforeAll` (one `new`, one `contacted`); seed an NA sales rep "Test Rep B" who has zero leads (so we can reassign to them and assert the row count goes to 1).

**Test 1 — Overview golden path:**
1. Login as NA country admin via `/api/e2e-login`.
2. Visit `/NA`.
3. Assert all five KPI tiles render and contain numbers (not "—" — seed data exists).
4. Assert "Leads by Service" card renders at least one bar.
5. Assert "Lead Status Pipeline" funnel renders 5 segments.
6. Assert "Sales Rep Performance" table renders at least one row including "Test Rep B" with `leads_assigned = 0`.
7. Assert "Speed to Lead" gauge renders an `%` number and the sparkline `<svg>` is non-zero in size (proves Recharts didn't 0×0 collapse — RESEARCH.md pitfall 2).

**Test 2 — Range picker drives Converted/Lost tiles:**
1. From `/NA`, mark one seeded lead `converted` via the queue RPC (or `mcp__supabase-paratusgroup__execute_sql`).
2. Reload `/NA?range=today` → "Converted" tile = 1.
3. Click range picker → "This week" → URL becomes `?range=week` → "Converted" tile still shows the converted lead (week ⊃ today).
4. Pick a custom past-only range that excludes today → "Converted" tile = 0.

**Test 3 — Reassignment + audit:**
1. Visit `/NA/leads`.
2. Filter to `status=new`.
3. Open ⋮ menu on the first lead → "Reassign…".
4. Pick "Test Rep B" → Save.
5. Assert toast appears (success).
6. Assert the lead's "Assigned To" cell now reads "Test Rep B".
7. Assert via `mcp__supabase-paratusgroup__execute_sql` that a `lead_events` row exists with `type='reassigned'` and `payload->>'to_agent_id'` = the test rep B's UUID.

**Test 4 — CSV export:**
1. Visit `/NA/leads`.
2. Filter to `status=new`.
3. Click "Export CSV" → wait for download event.
4. Read the downloaded file's body.
5. Assert: `Content-Type` header is `text/csv`, body's header row contains `id,full_name,email,phone_e164,status,form_slug,assigned_to,country_code,created_at,first_contacted_at,lost_reason`, every data row's `country_code` column is `NA`, every data row's `status` column is `new`.

**Test 5 — Cross-tenant block (defensive):**
1. Login as the BW country admin.
2. Visit `/NA` directly.
3. Assert middleware redirects to `/BW` OR the page renders with all-zero tiles for NA (RLS gives the BW user no NA rows). Whichever the middleware does — make the test match the actual behaviour, not the spec.
4. Try `/api/country-admin/export-leads?` → response either returns 0 NA rows or an empty CSV. Assert no NA rows present.

**Don't write fragile selectors** — use `data-testid` on the components. Add `data-testid` props in 04-02/04-03 plans where missing (look for `data-testid="date-range-picker-trigger"` precedent in `_components/date-range-picker.tsx` line 80) — extend the same pattern to `kpi-strip-tile-{key}`, `agent-performance-table`, `reassign-dialog-save`, `export-csv-link`, `lead-list-row-{id}`. If you find a missing testid during E2E authoring, ADD IT (small surgical edit to the component) — don't fight CSS-selector flakiness.

Set up the spec to run in CI eventually but for v1 it's a local-dev test (port 3012 prerequisite, same as Phase 3). Document this in `apps/web/e2e/README.md` if not already documented.
