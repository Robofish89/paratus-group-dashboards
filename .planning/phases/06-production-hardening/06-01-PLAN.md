<task type="auto">
  <name>Task 1: Harden production lead ingestion endpoint</name>
  <files>apps/web/app/api/leads/ingest/route.ts, packages/supabase/src/dal/leads.ts, .env.local.example, PRD/lead-ingestion.md</files>
  <action>
    Take the Phase 2 endpoint to production grade for the chosen path:
    1. Validate `X-Paratus-Secret` against `LEAD_INGEST_SHARED_SECRET` with constant-time compare (`timingSafeEqual` from `crypto`); reject with 401 on mismatch.
    2. Validate `X-Paratus-Signature` HMAC-SHA256 over the raw request body against `LEAD_INGEST_HMAC_SECRET`; reject with 401 on mismatch. Read raw body via `req.text()` BEFORE `req.json()` so signature verification matches the bytes the sender signed.
    3. Zod-parse the body; reject 422 with structured error on parse failure.
    4. Resolve `form_slug` and `country_code` against `forms` and `form_country_routing` tables; reject 422 with `unknown_form` / `unknown_country` codes if not found. Reject leads for countries with `status='coming_soon'` (return 422 with `country_not_active`).
    5. Idempotency: dedupe on `(form_slug, normalized_email_or_phone, submitted_at within 5 min window)`. Return 200 + `{ status: 'duplicate', lead_id }` if matched. Use a unique partial index on `leads(form_slug, dedupe_key, submitted_at_minute)` rather than a SELECT-then-INSERT (avoids race).
    6. Insert lead → insert `lead_events(type='created')` → invoke round-robin assignment function → insert `lead_events(type='assigned')`. Wrap in a single Postgres function `ingest_lead_with_assignment(payload jsonb)` so the whole insert+assignment is one transaction.
    7. Structured log on every request: `{ event: 'lead_ingest', lead_id, form_slug, country_code, latency_ms, dedupe_status, ip_hash }`. Use `console.log(JSON.stringify(...))` so Vercel log drains pick it up.
    8. Update `PRD/lead-ingestion.md` § "Phasing" with the chosen path and the operational runbook (how to rotate the secret, how to test from curl, how to debug a missed lead).
    9. Update `.env.local.example` with `LEAD_INGEST_SHARED_SECRET`, `LEAD_INGEST_HMAC_SECRET`, `RESEND_API_KEY`, `SLA_ALERT_FROM_EMAIL` (placeholder values, never real secrets).

    AVOID: catching errors and returning 200 (silent drop = data loss). Every failure path must return a non-2xx status with a structured error body.
  </action>
  <verify>
    - `npm run type-check` clean
    - `npm run lint` clean
    - Curl test from local with valid secret + HMAC returns 201; with bad secret returns 401; with bad HMAC returns 401; with unknown form returns 422; replay of same payload within 5 min returns 200 with `status: 'duplicate'`
    - `lead_events` shows `created` then `assigned` for the inserted lead
    - Vercel logs show one structured `lead_ingest` event per request
  </verify>
  <done>Endpoint accepts production traffic for the pilot country with HMAC + secret + idempotency + structured logging; PRD documents the chosen path and ops runbook; `.env.local.example` documents the new env vars.</done>

<task type="auto">
  <name>Task 2: SLA breach detection + Resend email alerts</name>
  <files>packages/supabase/migrations/006_sla_alerts.sql, packages/supabase/src/dal/sla.ts, packages/supabase/src/lib/email.ts, apps/web/app/api/cron/sla-check/route.ts, apps/web/vercel.json</files>
  <action>
    Build SLA breach detection that runs every minute and emails the country admin (and optionally the assigned agent) when a lead has been unanswered for ≥5 minutes.

    1. Migration `006_sla_alerts.sql`:
       - Add column `leads.sla_breach_alerted_at timestamptz` (null = not yet alerted; set after first email).
       - Create view `v_sla_breaches` returning leads where `status IN ('new','assigned')` AND `first_contacted_at IS NULL` AND `submitted_at < now() - interval '5 minutes'` AND `sla_breach_alerted_at IS NULL`.
       - RLS: leave view unsecured at the DB level (used by service role only); add policy on `leads` for the `update sla_breach_alerted_at` path so only service role can write it.
    2. `packages/supabase/src/lib/email.ts`:
       - Single `sendEmail({ to, subject, html, replyTo? })` wrapper around Resend SDK. `import 'server-only'`. Read `RESEND_API_KEY` and `SLA_ALERT_FROM_EMAIL` from env. Throw on missing env (fail fast — do not silently no-op).
       - Render a plain `slaBreachEmail({ lead, country, ageMinutes })` helper that returns `{ subject, html }` — no client-side dependency, just template strings.
    3. `packages/supabase/src/dal/sla.ts`:
       - `getOpenBreaches(): Promise<LeadBreach[]>` — queries `v_sla_breaches`.
       - `markBreachAlerted(leadId: string)` — updates `leads.sla_breach_alerted_at = now()`.
       - `import 'server-only'`.
    4. `apps/web/app/api/cron/sla-check/route.ts`:
       - GET handler. Verify `Authorization: Bearer ${process.env.CRON_SECRET}` (Vercel cron sends this).
       - Fetch open breaches. For each: look up country admin email(s); call `sendEmail`; on success, `markBreachAlerted`. Log structured event per breach.
       - Return JSON `{ checked: n, alerted: m }`.
    5. `apps/web/vercel.json` — add cron entry: `{ "crons": [{ "path": "/api/cron/sla-check", "schedule": "* * * * *" }] }`.
    6. Generate `CRON_SECRET` server-side (will set in Vercel env via `vercel env add` in Plan 03 ops); document in `.env.local.example`.

    AVOID: looping `sendEmail` without rate guard — if Resend rate-limits, batch in chunks of 10 with `Promise.allSettled`. AVOID re-alerting on the same lead — `sla_breach_alerted_at IS NULL` filter is the dedupe.
  </action>

    - Manual trigger of cron route locally with seeded breached lead → email sent (use Resend test domain) → `sla_breach_alerted_at` populated → second invocation does NOT re-alert
