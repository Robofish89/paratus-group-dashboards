<task type="auto">
  <name>Task 2: Wire Call Now → mark contacted → modal → complete/callback</name>
  <files>
    apps/web/app/(sales-rep)/_components/queue-view.tsx,
    apps/web/app/(sales-rep)/_components/queue-card.tsx,
    apps/web/app/(sales-rep)/_components/queue-action-bar.tsx,

  <action>
    The DAL functions from 03-01 are server-only (`server-only` import). The
    queue-view client component cannot call them directly — it goes through
    thin route handlers under `apps/web/app/api/queue/*`. Each route is
    cookie-session-authed (no HMAC bypass), validates body against the Zod
    schemas, calls the DAL function, returns the result.

    1. **Three route handlers**, all `runtime = 'nodejs'`, all gated by an
       agent-role check inside the handler (defense-in-depth even though
       middleware also enforces):
       - `POST /api/queue/contact` — body `{ lead_id }`, calls `markLeadContacted(lead_id)`, returns `{ lead_id, first_contacted_at }`
       - `POST /api/queue/complete` — body matches `completeCallInput` Zod schema, calls `completeCall(input)`, returns lead+status+outcome
       - `POST /api/queue/callback` — body matches `scheduleCallbackInput`, calls `scheduleCallback(input)` AND a follow-up `completeCall({ outcome: 'callback', notes })` so the call event is recorded too. Wrap in a single try/catch — both must succeed or surface error.

       Pattern: use `await createClient()` from `@repo/supabase/server` to
       carry the agent's cookie session into the RPC. Each handler:
       ```ts
       const supabase = await createClient();
       const { data: { user } } = await supabase.auth.getUser();
       if (!user) return NextResponse.json({ error: 'unauthorized' }, { status: 401 });
       // Validate body with Zod, surface 400 on failure
       // Call DAL, surface 200 / 500
       ```

    2. **Wire Call Now** in `queue-view.tsx`:
       - Card "Call Now" click → `onCallNow(lead)` prop
       - `queue-view.tsx` handles: POST `/api/queue/contact` → on success, optimistically flip lead.status to 'contacted' in local state, then open modal with that lead
       - Modal `onSubmit({outcome, notes, lost_reason, callback_at})`:
         - if `outcome === 'callback'` → POST `/api/queue/callback` `{lead_id, scheduled_for: callback_at, notes}`
         - else → POST `/api/queue/complete` `{lead_id, outcome, notes, lost_reason}`
         - On success: remove lead from `queue` state, prepend to `completed` state if it's now in a terminal status, bump appropriate stat counter, close modal
       - On error: leave modal open with inline error; don't roll back the optimistic 'contacted' status (it actually flipped server-side on the contact call).

    3. **`queue-action-bar.tsx`** — small client component owning the modal
       state machine for one selected lead. Pulled out of `queue-view.tsx` so
       the page-level component stays under ~150 lines.

    Keep optimistic updates simple: server is authoritative via realtime.
    When the broadcast fires for the same lead (status update), the merge in
    `prependOrUpdate` from 03-02 either matches the optimistic state or
    overwrites it. No conflict resolution needed beyond "newest wins."

    Stub removal: delete the `console.log('call', lead.id)` placeholder from 03-02.
  </action>
