  <action>
The provisioning pattern is identical to Plan 07-03 Task 2e — only the `--country=<CC>` flag changes. Roll out one country per execution unit (don't batch — keeps the per-country cutover-checklist sign-off coherent with the actual work).

For each of: AO (Angola), BW (Botswana), CD (DRC), SZ (Eswatini), KE (Kenya), NA (Namibia), RW (Rwanda), ZA (South Africa), TZ (Tanzania), UG (Uganda), ZM (Zambia):

**Per-country sequence:**
1. **Confirm contact list with William** — verify the rows for `<CC>` in `.planning/rollout-contacts.csv` are accurate. If William hasn't yet supplied this country's contact list, pause that country and continue with the next one.
2. **Run the script:** `npx tsx apps/web/scripts/provision-users.ts --country=<CC> .planning/rollout-contacts.csv`. Expected: `event:'user_provisioned'` × N rows + `event:'provision_summary'` `failed: 0`. If failed > 0, debug per the script's structured error log; do not flip the form-side webhook for this country until provisioning is clean.
3. **Smoke test from one seat per role** — log into the country admin seat (the one whose welcome email you just sent), assert dashboard loads at `/<cc-slug>`, KPIs render zeros, agents leaderboard shows the provisioned agents. Optionally also smoke from one agent seat — repeat the Plan 07-03 Task 3 pattern.
4. **Cross-tenant negative test** — from this new country admin seat, attempt `/<other-cc-slug>` for any other active country. Assert: 403/route-protect.
5. **Flip the form-side ingestion** for `<CC>`. Path 1 or Path 2 per the per-country decision in `07-USER-SETUP.md` (default Path 2; if William signalled Path 1 for any specific country, honour that).
6. **Observe first real lead** within 30 min, OR submit a synthetic lead via the live form. Confirm realtime arrival in the agent's queue.
7. **Tick CUTOVER.md `<CC>` section** — items 1-8, plus a 4-h-soak observation note (we don't gate the next country on a 24-h soak — that was the pilot's job; subsequent countries gate on a 4-h smoke window where smoke + first-lead + zero P1/P2 + audit captured is enough).
8. **William sign-off line** for `<CC>` — date and signature in CUTOVER.md.
9. **Move to next country.**

**Cadence target:** ~30 min per country including provisioning + smoke + form-side flip + 4-h smoke window. Eleven countries × 30 min = a focused day's work, but realistically spread across William's available windows for sign-off.

**If a country fails the 4-h smoke window:**
- 429s on ingest → check Upstash rate-limit cap; if Paratus's form throughput per country exceeds 60 req/min, flag for retainer-scope rate-limit retuning (default cap was sized for the pilot's expected volume).
- Cross-tenant leak observed → STOP all rollouts. RLS regression is a Phase 1 invariant — escalate to a hotfix plan.
- Sentry P1/P2 → triage; the bar is "no unresolved P1/P2 by sign-off time".
- Lead never arrives → check the form-side webhook is hitting the correct URL with the production HMAC; check `/api/leads/ingest` Vercel logs for HMAC validation failures.

AVOID:
  - Batch-provisioning multiple countries in a single script invocation. The `--country` filter exists precisely to keep the work per-country attributable.
  - Skipping the cross-tenant negative test for "non-pilot" countries. Every country flip is a fresh RLS contract; we re-prove it every time.
  - Pre-checking CUTOVER.md boxes optimistically. The boxes are evidence — they get checked when the work is done.
  - Lowering the bar for non-pilot countries. The pilot got a 24-48 h soak because it's the first; the others get a 4 h smoke + first-lead because the pilot proved the system. The bar is the same SHAPE (smoke + first-lead + signed off), just compressed.
