<objective>
Build the bulk-invite engine for Phase 7 rollout — a branded React Email invite template, a `sendInviteEmail` Resend wrapper, and an idempotent `provision-users.ts` script that takes a CSV of `{email, role, country_code, full_name}`, creates each user via the Auth admin API, upserts the `user_roles` row BEFORE the invite is consumed (avoiding the JWT-hook race), and sends a hand-rolled invite via Resend (avoiding Supabase's 2/hr default-SMTP cap and the broken re-invite path on `inviteUserByEmail`). Backed by a vitest integration test against the hermetic local Supabase stack from Phase 6.

Purpose: Bulk-provisioning ~30 humans across 12 countries demands a script that survives re-runs (lost emails, expired tokens, partial failures) without leaking credentials, double-sending, or creating users without role rows. This plan ships the engine; Plan 07-03 actually runs it for the pilot country.

Output:
- `packages/supabase/src/lib/emails/invite.tsx` — React Email template (paratus-blue + accent-orange, mirrors the SLA-breach template patterns from Phase 6)

- `apps/web/scripts/__tests__/provision-users.test.ts` — vitest integration test against local Supabase

  <action>
Add `tsx` as a devDependency to `apps/web/package.json` (latest 4.x — `tsx@^4.19.0` is the current stable line). Use `npm install --workspace=apps/web --save-dev tsx` so the workspace lock stays consistent.

Create `apps/web/scripts/provision-users.ts` matching the sketch in `07-RESEARCH.md` Pattern 1, with these execution-tightened details:
  - **Module shape:** plain Node script (no `'use server'`, no `server-only` import — this runs from a developer machine, not a Next.js runtime).
  - **Imports:** `createAdminClient` from `@repo/supabase/admin`; `sendInviteEmail` from `@repo/supabase/lib/email`; `parse` from `csv-parse/sync` (already a transitive dep via `papaparse`'s peer chain — confirm with `npm ls csv-parse` and add directly to `apps/web` devDeps if not resolvable).
  - **CLI argv:** first positional argument = path to CSV (default `.planning/rollout-contacts.csv`); flag `--dry-run` parses and validates the CSV without making any Supabase or Resend calls; flag `--country=<CC>` filters the CSV to a single country (used by Plan 07-03 for the Mozambique-only first run).
  - **Env contract:** asserts `SUPABASE_SERVICE_ROLE_KEY`, `NEXT_PUBLIC_SUPABASE_URL`, `RESEND_API_KEY`, `NEXT_PUBLIC_APP_URL` present at startup; fails fast with a single line listing missing vars.
  - **CSV row shape:** `{ email, role, country_code, full_name }`. Validate per row with a Zod schema (inline — small, no need for a `@repo/supabase/schemas` entry). Empty `country_code` translates to `null` for HQ admins (schema already requires `country_code IS NULL` when role = `hq_admin`). Reject rows with `role='hq_admin' AND country_code != ''` and `role IN ('country_admin','agent') AND country_code = ''` BEFORE any Supabase call.
  - **Per-row flow** (in this exact order — DO NOT reorder, the JWT hook race depends on it):
    1. `admin.auth.admin.createUser({ email, email_confirm: false, user_metadata: { full_name } })`. On `code === 'email_exists'` (or message containing "already been registered"), call `admin.auth.admin.listUsers({ page: 1, perPage: 1000 })` and look up the existing user by email. Throw if neither path yields a user_id.
    2. `admin.from('user_roles').upsert({ user_id, role, country_code: country_code || null, is_active: true }, { onConflict: 'user_id' })`. This is the line that closes the JWT-hook race.
    3. `admin.auth.admin.generateLink({ type: 'invite', email, options: { redirectTo: `${process.env.NEXT_PUBLIC_APP_URL}/auth/accept-invite` } })`. Returns `data.properties.action_link`.
    4. `sendInviteEmail({ to: email, fullName, role, countryName: countryNameLookup(country_code), actionUrl: link, userId, supportEmail: 'william@brainstormprojects.co' })`. We pin a literal default support contact for v1 — Phase 7 retainer can wire it through env later.
  - **Logging:** every successful row prints a single `JSON.stringify({ event: 'user_provisioned', email, role, country_code, ts: new Date().toISOString() })`; failures print `event: 'user_provision_failed'` with `email, message, step` (where `step` is one of `create | role_upsert | generate_link | send_email`). One line per row, JSON-only — feeds the Vercel runtime drain pattern Phase 6 standardised.
  - **Idempotency:** re-running against the same CSV produces zero net writes (createUser → email_exists → listUsers lookup; upsert is no-op when row matches; generateLink re-issues OTP; Resend re-sends — that last one IS a re-send, which is the desired behaviour for "user lost the email").
  - **Exit codes:** `0` if every row succeeded, `1` if any row failed (so a CI/CD invocation surfaces failure). Print a one-line summary: `{ event: 'provision_summary', total, succeeded, failed }`.

  - Using `inviteUserByEmail` anywhere. Open Supabase issue [#2180](https://github.com/supabase/auth/issues/2180) breaks re-invite for existing users; the `generateLink + Resend` path is the documented workaround.
  - Sending invites in parallel via `Promise.all`. Resend free-tier ratelimit is 10 req/sec — sequential `for…of` (~30 rows × 1s = trivial) keeps us well under and makes failure attribution per-row trivial.
  - Concatenating the secret into the log line (the script handles secrets only via env reads; never log `process.env.RESEND_API_KEY` or the service-role key, even partial).

Read it from the same source the runtime uses: `apps/web/app/(country-admin)/[country]/_lib/country.ts` exports a `countryNameByCode` map; import that or extract a tiny copy if importing from the Next route causes a transitive dependency that breaks tsx.
