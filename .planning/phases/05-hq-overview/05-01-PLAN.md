Create `packages/supabase/migrations/00013_hq_overview.sql`. Header comment: `-- 00013_hq_overview: views + RPC + broadcast trigger for the HQ Overview dashboard (Phase 5)`. Define the surfaces below in this exact order — every view ends with `WITH (security_invoker = true)` and `GRANT SELECT ON ... TO authenticated`. The RPC is `SECURITY DEFINER` with `EXECUTE` granted to `authenticated` only. **No view body reads `(SELECT auth.jwt() ->> 'country_code')`** — HQ admin's claim is NULL there, so reading it returns no rows (RESEARCH.md pitfall 1).

1. **`group_today_stats`** — single-row group-wide rollup powering the 5-tile KPI strip. Columns:
   - `active_country_count bigint` — `count(c.code) FILTER (WHERE c.status = 'active')`
   - `total_leads_group bigint` — all-time count of `leads` across active countries (matches mockup "Total Leads (Group)" 8,432 — all-time)
   - `new_today_group bigint`, `contacted_today_group bigint`, `converted_today_group bigint`, `lost_today_group bigint` — sums of the matching columns from `country_today_stats` (00011), filtered to active countries
   - `conversion_rate_alltime numeric(5,4)` — `count(l.id) FILTER (WHERE l.status='converted')::numeric / NULLIF(count(l.id), 0)` over active countries (decision per RESEARCH.md open question 3: all-time, matches mockup math)
   - `avg_speed_to_lead_seconds_today numeric(10,2)` — `avg(EXTRACT(EPOCH FROM (l.first_contacted_at - l.submitted_at)))` filtered to leads with `first_contacted_at IS NOT NULL` AND `created_at >= start_of_today_in_country_tz`. Compute the today-boundary per country via `(date_trunc('day', now() AT TIME ZONE c.timezone) AT TIME ZONE c.timezone)` and join. **No JWT reads.**

   Shape: `FROM public.countries c LEFT JOIN public.country_today_stats t ON t.country_code = c.code LEFT JOIN public.leads l ON l.country_code = c.code WHERE c.status = 'active'`. Returns exactly one row (no GROUP BY needed — pure aggregations).

2. **`country_performance_today`** — per-country leaderboard, mockup-shape (NOT an extension of `country_leaderboard` from 00006; that view is 30d-windowed, kept for trend context). One row per active country. Columns:
   - `country_code text` (FK), `country_name text`
   - `total_leads bigint` — all-time count for that country
   - `new_today bigint` — leads created today in country tz (`date_trunc('day', now() AT TIME ZONE c.timezone) AT TIME ZONE c.timezone` boundary, half-open `[start, start + 1 day)`)
   - `contacted_pct numeric(5,4)` — `count(l.id) FILTER (WHERE l.first_contacted_at IS NOT NULL)::numeric / NULLIF(count(l.id), 0)` (all-time)
   - `converted_pct numeric(5,4)` — `count(l.id) FILTER (WHERE l.status='converted')::numeric / NULLIF(count(l.id), 0)` (all-time)
   - `avg_response_seconds numeric(10,2)` — avg seconds-to-first-contact, filtered to `first_contacted_at IS NOT NULL` (all-time — matches mockup; today-only would be too volatile across small-volume countries)

   `FROM public.countries c LEFT JOIN public.leads l ON l.country_code = c.code WHERE c.status = 'active' GROUP BY c.code, c.name ORDER BY count(l.id) DESC`.

3. **`leads_by_service_group`** — all-time count per `form_slug` across active countries, powering the "Leads by Service (Group)" horizontal-bar list. One row per form_slug. Columns: `form_slug text`, `leads_count bigint`. Decision per RESEARCH.md open question 1: **all-time** (mockup bars sum to "Total Leads (Group)" 8,432). Document the divergence from `leads_by_service_today` (00011, today-only per-country) in a SQL comment.

   `FROM public.leads l JOIN public.countries c ON c.code = l.country_code WHERE c.status = 'active' GROUP BY l.form_slug ORDER BY count(l.id) DESC`.

4. **`group_speed_to_lead_series(p_days int DEFAULT 7)` RPC** — per-day group-wide P50/P75 of seconds-to-first-contact, last `p_days` days. Mirror the existing `speed_to_lead_series(text, timestamptz, timestamptz)` RPC body (00011, lines 413–474) but: (a) parameter is `p_days int` not country+range; (b) day boundary uses UTC (`date_trunc('day', l.created_at)::date`) — group view spans tz so per-country tz makes no sense; (c) role guard accepts `hq_admin` only (raise `forbidden_role` 42501 otherwise — country admins have no business reading the group series). Returns `(day date, median_seconds numeric(10,2), p75_seconds numeric(10,2))`. `LANGUAGE plpgsql SECURITY DEFINER SET search_path = public, pg_temp`. `REVOKE ALL ... FROM PUBLIC, anon; GRANT EXECUTE ... TO authenticated`. Filter: `l.first_contacted_at IS NOT NULL AND l.created_at >= now() - (p_days || ' days')::interval`.

5. **`broadcast_lead_to_group()` trigger function + `leads_broadcast_group` trigger** — mirror `broadcast_lead_to_country()` from 00008 (lines 71–95) but with topic `'group:all'` and no per-country filter. `AFTER INSERT OR UPDATE ON public.leads FOR EACH ROW`. Listening pattern is `event:'*'` not `'INSERT'` (RESEARCH.md don't-hand-roll table — webhook path emits `UPDATE` when `assign_lead` flips `assigned_to`, not `INSERT`).

6. **`hq_group_topic` RLS policy on `realtime.messages`** — `FOR SELECT TO authenticated USING ((SELECT auth.jwt() ->> 'user_role') = 'hq_admin' AND realtime.topic() = 'group:all')`. Drop-and-recreate idempotent. The existing per-country `hq_country_topic` policy from 00008 stays — HQ retains the ability to subscribe to a specific `country:<code>` topic when drilling in.

**Apply:** Use `mcp__supabase-paratusgroup__apply_migration` with `name: "hq_overview"` and the file body. Verify with `mcp__supabase-paratusgroup__list_migrations`.

**Regenerate types:** Run `mcp__supabase-paratusgroup__generate_typescript_types` and write the result to `packages/supabase/src/types/database.ts` (full overwrite — same path Phase 4 plan 04-02 used). Run `npm run type-check` from repo root to confirm no breakage.

**Security advisor:** Run `mcp__supabase-paratusgroup__get_advisors --type security` and `--type performance`. Zero new warnings introduced by 00013. Pre-existing warnings stay tracked into Phase 6.
