    1. **Migration `packages/supabase/migrations/00015_audit_log.sql`** — apply via `mcp__supabase-paratusgroup__apply_migration`:
       - Table:
         ```sql
         CREATE TABLE audit_log (
           id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
           actor_id        uuid REFERENCES auth.users(id),
           actor_role      text NOT NULL,        -- 'agent' | 'country_admin' | 'hq_admin' | 'system'
           country_code    text NOT NULL,        -- the country the action targets
           action          text NOT NULL,        -- e.g. 'lead.reassign', 'lead.complete', 'lead.no_answer'
           target_type     text NOT NULL,        -- 'lead' | 'user_role' | 'callback'
           target_id       text NOT NULL,
           diff            jsonb NOT NULL,       -- changed-fields-only; { field: { before, after } }
           visible_to_country_codes text[] NOT NULL,  -- handles HQ-initiated cross-country reassign (writes one row visible to source AND target country admins)
           created_at      timestamptz NOT NULL DEFAULT now(),
           ip_hash         text                  -- sha256(IP) — never raw IP
         );
         CREATE INDEX audit_log_country_created_idx ON audit_log (country_code, created_at DESC);
         CREATE INDEX audit_log_target_idx          ON audit_log (target_type, target_id);
         CREATE INDEX audit_log_visible_gin_idx     ON audit_log USING gin (visible_to_country_codes);
         ```
       - RLS enabled with `(SELECT …)` initPlan caching + `TO authenticated` (per Phase 6 RESEARCH pattern 1):
         ```sql
         ALTER TABLE audit_log ENABLE ROW LEVEL SECURITY;

         CREATE POLICY "hq sees all audit_log"
           ON audit_log FOR SELECT TO authenticated
           USING ((SELECT auth.jwt() ->> 'user_role') = 'hq_admin');

         CREATE POLICY "country admin sees scoped audit_log"
           ON audit_log FOR SELECT TO authenticated
           USING (
             (SELECT auth.jwt() ->> 'user_role') = 'country_admin'
             AND (SELECT auth.jwt() ->> 'country_code') = ANY (visible_to_country_codes)
           );
         ```
         — no INSERT/UPDATE/DELETE policies → only the SECURITY DEFINER RPC + service-role can write. Agents see zero rows by design (no policy applies to `user_role='agent'`).
       - `record_audit` RPC:
         ```sql
         CREATE OR REPLACE FUNCTION record_audit(
           p_action      text,
           p_target_type text,
           p_target_id   text,
           p_country_code text,
           p_diff        jsonb,
           p_visible_to_country_codes text[] DEFAULT NULL,
           p_ip_hash     text DEFAULT NULL
         ) RETURNS uuid
         LANGUAGE plpgsql SECURITY DEFINER SET search_path = public, pg_temp AS $$
         DECLARE v_id uuid;
         BEGIN
           INSERT INTO audit_log (actor_id, actor_role, country_code, action, target_type, target_id, diff, visible_to_country_codes, ip_hash)
           VALUES (
             auth.uid(),
             COALESCE(auth.jwt() ->> 'user_role', 'system'),
             p_country_code,
             p_action, p_target_type, p_target_id, p_diff,
             COALESCE(p_visible_to_country_codes, ARRAY[p_country_code]),
             p_ip_hash
           )
           RETURNING id INTO v_id;
           RETURN v_id;
         END $$;

         REVOKE EXECUTE ON FUNCTION record_audit FROM PUBLIC, anon;
         GRANT  EXECUTE ON FUNCTION record_audit TO authenticated;
         ```
       - The RPC runs as `authenticated`, not service-role, so `auth.uid()` and `auth.jwt()` resolve to the calling agent/admin (SECURITY DEFINER bypasses RLS but session context survives).
       - Migration header explains the design: `visible_to_country_codes` is an ARRAY because HQ-initiated cross-country reassignment must be visible to BOTH source and target country admins (not just the target). For a same-country action, the array is `[country_code]`. For cross-country reassign, it's `[source_country, target_country]`.

    3. **DAL `packages/supabase/src/dal/audit.ts`** (`import 'server-only'`):
       - Type `AuditAction = 'lead.reassign' | 'lead.complete' | 'lead.callback' | 'lead.no_answer' | 'lead.contact' | 'user_role.update'` (extend later as new write paths land).
       - Type `AuditTarget = 'lead' | 'user_role' | 'callback'`.
       - `recordAudit({ action, targetType, targetId, countryCode, diff, visibleToCountryCodes?, ipHash? })` — uses the **session-authed** `createClient()` (cookie-bound), NOT the admin client. SECURITY DEFINER on the RPC bypasses RLS but `auth.uid()` resolves to the calling user. Throw on RPC error.
       - `getAuditLog({ countryCode?, page, pageSize, filter? }): Promise<{ rows: AuditRow[]; total: number }>` — uses the **session-authed** client so RLS scopes results by role. Country admins implicitly see only their country (RLS does the work); HQ passing `countryCode` filters on the column. Pagination is offset-based for v1 (matches Phase 4's lead list approach; cursor migration tracked alongside Plan 06-04 if it lands together).
       - `hashIpAddress(ip: string): string` — `crypto.createHash('sha256').update(ip + IP_HASH_SALT).digest('hex')` where `IP_HASH_SALT` is a per-deploy constant pulled from env. Document in `.env.local.example` and 06-USER-SETUP.md.
       - Helper `computeDiff<T extends Record<string, unknown>>(before: T, after: T): jsonb` — returns `{ field: { before, after } }` only for changed fields. Avoids storing whole-row snapshots.

    AVOID: storing the full `before`/`after` row in `diff` (PII surface, index bloat); writing audit rows from a non-session client (loses `auth.uid()`); using IP directly without hashing (PII compliance). AVOID a separate `audit_log_can_view` predicate function — keep RLS in the policy USING clause for InitPlan caching.
